## Audio Stimuli in the Behavioral Task

### 1. Go Cue Tone

The primary audio stimulus in the IBL 2-alternative forced choice (2AFC) task is the **go cue tone**.

**Characteristics:**
- **Frequency**: 5 kHz sine wave
- **Duration**: 100 milliseconds
- **Timing**: Occurs approximately at stimulus onset (when visual grating appears)
- **Purpose**: Signals to the animal that they can make a response
- **Synchronization**: Generated by Xonar soundcard with TTL sync pulse fed back to Bpod

**From trials metadata** (`_metadata/trials.yml`):
```yaml
goCue_times:
  name: go_cue_time
  description: |
    The start time of the go cue tone. This is the time the sound is
    actually played, that is, the command sent through soundcard sync
    was fed back into Bpod. The go cue tone is a 100ms 5 kHz sine
    wave and occurs approximately at stimulus onset.
```

**Scientific Significance:**
- Provides a consistent auditory cue across trials
- Serves as a synchronization marker between behavioral and neural recording systems
- Allows alignment of behavioral events with electrophysiology data

### 2. Feedback Audio

Audio feedback is provided to the animal based on their behavioral response:

**Positive Feedback (Correct Trials):**
- Water reward delivery (solenoid valve TTL trigger)
- No sound component (only the sound of water dispensing)

**Negative Feedback (Incorrect Trials/Timeouts):**
- White noise burst
- Indicates error or timeout

**From trials metadata** (`_metadata/trials.yml`):
```yaml
feedback_times:
  name: feedback_time
  description: |
    The time of feedback delivery. For correct trials this is the
    time of the valve TTL trigger command, for incorrect trials this
    is the time of the white noise output trigger.
```

## Audio Recording System

### Hardware Setup

**Microphone Recording:**
- **Device**: Microphone integrated with Xonar sound system
- **Raw Format**: WAV files (`_iblrig_micData.raw.wav`)
- **Compressed Format**: FLAC files (`_iblrig_micData.raw.flac`)
- **Synchronization**: TTL pulse from soundcard fed back to Bpod for precise timing

**Sound Generation:**
- **System**: Xonar soundcard
- **Output**: Go cue tones, white noise for error feedback
- **Sync**: TTL pulses sent to Bpod to mark audio onset times

### Recording Pipeline

The audio recording pipeline consists of two main stages:

#### 1. Audio Compression (`AudioCompress` task)

**Location**: `ibllib/ibllib/pipes/audio_tasks.py`

Converts raw WAV files to FLAC format for efficient storage:

```python
command = "ffmpeg -i {file_in} -y -nostdin -c:a flac -nostats {file_out}"
# Converts _iblrig_micData.raw.wav to _iblrig_micData.raw.flac
```

**Benefits:**
- Lossless compression (no audio quality loss)
- Significant file size reduction (~50% smaller)
- Maintains full temporal resolution

#### 2. Audio Synchronization (`AudioSync` task)

**Location**: `ibllib/ibllib/pipes/audio_tasks.py`

Extracts audio events and synchronization markers:

**Generated Output Files:**
- `_iblmic_audioOnsetGoCue.times_mic.npy` - Go cue detection times from microphone
- `_iblmic_audioSpectrogram.frequencies.npy` - Frequency axis for spectrogram
- `_iblmic_audioSpectrogram.power.npy` - Power spectral density
- `_iblmic_audioSpectrogram.times_mic.npy` - Time axis for spectrogram

**Current Implementation:**
- Designed for Bpod-controlled experiments with Xonar sound system
- Extracts go cue onset times by detecting 5 kHz tone
- Generates time-frequency spectrograms for visualization

## Audio Processing Pipeline

### Go Cue Detection

**Location**: `ibllib/ibllib/io/extractors/training_audio.py`

The `detect_ready_tone()` function identifies go cue tones in the audio recording:

**Detection Algorithm:**
1. Apply bandpass filter around 5 kHz
2. Compute Hilbert envelope to extract amplitude
3. Identify peaks exceeding threshold (0.8 ratio)
4. Extract onset times

**Parameters:**
- **Target frequency**: 5 kHz (go cue tone)
- **Detection threshold**: 0.8 (Hilbert envelope ratio)
- **Time tolerance**: ±200ms deviation allowed between audio and Bpod detection

### Spectrogram Computation

**Location**: `ibllib/ibllib/io/extractors/training_audio.py` (`welchogram` function)

For long audio recordings, the pipeline uses Welch's method for efficient spectral analysis:

**Method:**
1. Split audio into overlapping windows (2^18 samples, ~262k samples)
2. Compute Welch periodogram for each window (512-point FFT)
3. Apply Hann window function
4. 50% overlap between consecutive windows
5. Convert power to dBFS (decibels relative to full scale)

**Output:**
- Time-frequency matrix showing power spectral density
- Frequency resolution: ~1-2 Hz bins
- Temporal resolution: depends on window size and overlap

**SPL Calibration:**
- Reference level: 85 dB SPL for go cue tone (5 kHz)
- Allows conversion from arbitrary units to sound pressure level

### Synchronization with Behavioral Data

**Location**: `ibllib/ibllib/io/extractors/training_audio.py` (`extract_sound` function)

The audio extraction pipeline aligns microphone recordings with Bpod behavioral timestamps:

**Process:**
1. Load raw audio file (FLAC or WAV)
2. Compute spectrogram using Welch's method
3. Detect go cue tones at 5 kHz
4. Load Bpod go cue times from trials data
5. Align microphone times with Bpod times
6. Validate alignment (check for ±200ms tolerance)

**Why This Matters:**
- Bpod records behavioral events with microsecond precision
- Microphone provides independent verification of audio timing
- Cross-validation ensures synchronization accuracy
- Detects potential timing issues or hardware failures

## Passive Task Audio

### Overview

After the main 2AFC task, sessions include a **passive replay period** where stimuli are presented without requiring a behavioral response.

**Documentation**: `documentation/ibl_passive_task.md`

### Audio Component in Passive Task

**Stimuli Presented:**
- **Go Cue Tones**: Same 5 kHz tones used in the main task
- **White Noise Bursts**: Error feedback sounds from the task

**Purpose:**
- Establish baseline neural responses to auditory stimuli
- Compare task-engaged vs. passive auditory processing
- Study attention-dependent modulation of sensory responses
- Control for sensory vs. motor/decision-related activity

### Data Files

**Dataset**: `_ibl_passiveStims.table.csv`

Contains information about audio stimuli during passive replay:
- Stimulus type (tone vs. noise)
- Onset times
- Duration
- Intensity parameters

**Availability**: ~68% of Brain Wide Map sessions include passive task data

## Available Datasets via ONE API

### Audio Datasets

The following audio-related datasets can be accessed using the ONE (Open Neurophysiology Environment) API:

#### Raw and Compressed Audio

```python
# Raw compressed microphone recording
'_iblrig_micData.raw.flac'  # FLAC format, lossless compression
```

**Usage:**
```python
from one.api import ONE
one = ONE()

# Download raw audio file
filename = one.load_dataset(eid, '_iblrig_micData.raw.flac', download_only=True)
```

#### Processed Audio Features

```python
# Spectrogram data
'_iblmic_audioSpectrogram.frequencies'  # Frequency axis (Hz)
'_iblmic_audioSpectrogram.power'        # Power spectral density (dBFS)
'_iblmic_audioSpectrogram.times_mic'    # Time axis (seconds)

# Go cue detection
'_iblmic_audioOnsetGoCue.times_mic'     # Detected go cue onset times
```

**Usage:**
```python
# Load spectrogram object
TF = one.load_object(eid, 'audioSpectrogram', collection='raw_behavior_data')

# Access fields
frequencies = TF['frequencies']  # Hz
power = TF['power']              # dBFS
times = TF['times_mic']          # seconds
```

#### Passive Task Audio

```python
# Passive stimuli table
'_ibl_passiveStims.table.csv'  # Audio events during passive replay
```

### Example: Plotting Audio Spectrogram

**Location**: `ibllib/examples/one/behavior/plot_microphone_spectrogram.py`

```python
from one.api import ONE
import matplotlib.pyplot as plt

one = ONE()
eid = 'your-experiment-id'

# Load spectrogram
TF = one.load_object(eid, 'audioSpectrogram', collection='raw_behavior_data')

# Plot
plt.pcolormesh(TF['times_mic'], TF['frequencies'], TF['power'].T,
               shading='auto', cmap='viridis')
plt.ylabel('Frequency (Hz)')
plt.xlabel('Time (s)')
plt.colorbar(label='Power (dBFS)')
plt.show()
```

## Audio in NWB Conversion

### What IS Included in NWB Files

While raw audio recordings are not directly converted to NWB format, audio-related information is preserved in several ways:

#### 1. Trials Table - Audio Event Timing

**Location**: `IBL-to-nwb/src/ibl_to_nwb/datainterfaces/_brainwide_map_trials_interface.py`

The trials table includes audio-related columns:

| Column | Description | Units |
|--------|-------------|-------|
| `goCue_times` | Time when go cue tone (5 kHz, 100ms) was played | seconds |
| `feedback_times` | Time of feedback delivery (water valve or error noise) | seconds |
| `feedbackType` | Type of feedback: +1 (correct/water), -1 (error/noise) | categorical |

**NWB Representation:**
- Stored as columns in `nwbfile.trials` DynamicTable
- Times are referenced to session start time
- Allows alignment with neural data

#### 2. NIDQ Digital Channels - Audio System TTLs

**Location**: `IBL-to-nwb/src/ibl_to_nwb/datainterfaces/_ibl_nidq_interface.py`

Audio-related digital signals recorded by the National Instruments DAQ (NIDQ) board:

**Channels:**
- Soundcard sync pulses (go cue triggers)
- Audio system TTL events
- Feedback tone triggers

**NWB Storage:**
- Digital events stored as timestamps in processing module
- Device-specific naming (e.g., "audio_sync", "sound_trigger")
- Synchronized with electrophysiology timestamps via NIDQ clock

**Important Note:**
For some sessions, audio was not wired into the FPGA. The IBL team recovered these using Bpod-to-FPGA time interpolation (documented in 2025 Q1 data release notes).

#### 3. Passive Task Events

**Location**: Passive stimuli information included in trials or stimulus tables

Audio events during passive replay periods are stored with:
- Event type (tone, noise)
- Onset times
- Stimulus parameters

### What is NOT Included in NWB Files

The following audio data are NOT directly converted to NWB:

- Raw microphone recordings (`.flac` files)
- Audio spectrograms (frequency-time representations)
- Continuous audio waveforms
- Audio power spectral density matrices

**Rationale:**
1. **File size**: Raw audio files are large and don't compress efficiently in HDF5 format
2. **Scientific relevance**: The critical information (timing of auditory events) is captured in the trials table
3. **Separate access**: Researchers interested in acoustic analysis can access `.flac` files via ONE API
4. **Storage efficiency**: Avoiding redundancy between ONE and NWB storage systems

### Accessing Full Audio Data

For analyses requiring raw audio or spectrograms:

1. **Use ONE API** to download audio files separately:
   ```python
   audio_file = one.load_dataset(eid, '_iblrig_micData.raw.flac')
   spectrogram = one.load_object(eid, 'audioSpectrogram')
   ```

2. **Use NWB files** for audio event timing and alignment with neural data:
   ```python
   nwbfile = NWBHDF5IO('file.nwb', 'r').read()
   go_cue_times = nwbfile.trials['goCue_times'][:]
   feedback_times = nwbfile.trials['feedback_times'][:]
   ```

3. **Combine both** for comprehensive analysis:
   - NWB for precise event times and neural alignment
   - ONE for raw audio, spectrograms, and acoustic features

## Technical Specifications

### Audio Recording Specifications

| Parameter | Value |
|-----------|-------|
| **Recording format** | WAV (raw), FLAC (compressed) |
| **Compression method** | FLAC via ffmpeg (lossless) |
| **Microphone system** | Xonar sound system with TTL sync |
| **Sampling rate** | Extracted from WAV file header (typically 44.1 or 48 kHz) |
| **Bit depth** | 16-bit (typical) |
| **File size (compressed)** | ~50% of raw WAV size |

### Go Cue Tone Specifications

| Parameter | Value |
|-----------|-------|
| **Frequency** | 5 kHz (pure sine wave) |
| **Duration** | 100 milliseconds |
| **Intensity** | 85 dB SPL (calibrated reference) |
| **Waveform** | Sine wave |
| **Rise/fall time** | Not specified (likely 5-10ms to avoid clicks) |

### Spectral Analysis Parameters

| Parameter | Value |
|-----------|-------|
| **Window function** | Hann window |
| **Window size** | 2^18 samples (~262,144 samples, ~5.5s at 48kHz) |
| **FFT points** | 512 samples |
| **Overlap** | 50% between consecutive windows |
| **Frequency resolution** | ~1-2 Hz bins |
| **Output units** | dBFS (decibels relative to full scale) |

### Synchronization Specifications

| Parameter | Value |
|-----------|-------|
| **Sync method** | Soundcard TTL pulse to Bpod |
| **Bpod timing precision** | Microsecond resolution |
| **Alignment tolerance** | ±200ms allowed deviation |
| **Validation** | Cross-check microphone vs. Bpod go cue times |
| **Failure handling** | Sessions with sync issues flagged in QC |

## Data Availability

### Brain Wide Map 2025 Data Release

**Audio-related data availability** across 459 BWM sessions:

| Data Type | Availability | Notes |
|-----------|-------------|-------|
| **Audio event times (trials)** | 100% (459/459) | goCue_times, feedback_times in trials table |
| **NIDQ audio TTLs** | 100% (459/459) | Audio sync pulses in digital channels |
| **Raw microphone recordings** | ~68% empirical | Only sessions with microphone hardware |
| **Audio spectrograms** | ~68% empirical | Computed from microphone recordings |
| **Passive task audio** | ~68% empirical | Subset of sessions with passive protocol |

### Known Issues and Patches

**2025 Q1 Audio Sync Patch:**

From the 2025 Brain Wide Map data release notes:

> "For a number of important ephys sessions the audio was somehow not wired into the FPGA, however everything else was present and the Bpod recorded these TTLs so we decided to use the bpod2fpga interpolation to recover the audio TTLs in FPGA time. These were then added to the _spikeglx_sync object and the trials were re-extracted."

**Impact:**
- Some sessions had audio synchronization recovered via interpolation
- Audio event times in these sessions are derived rather than directly measured
- Quality control flags may indicate interpolated sync
- Does not affect behavioral (Bpod) timing, only FPGA-aligned neural timing

### Quality Control

**Audio QC Checks:**
1. Microphone file existence and integrity
2. Go cue detection success rate
3. Alignment between microphone and Bpod times
4. Spectrogram quality and frequency coverage
5. Audio-FPGA synchronization validation

**Access QC Status:**
```python
# Check if audio passed QC
qc = one.load_dataset(eid, '_ibl_experiment.qc')
audio_qc = qc['audioQC']  # 'PASS', 'WARNING', or 'FAIL'
```

## Summary

The IBL Brain Wide Map experiments use audio stimuli primarily as behavioral cues (go cue tone, error feedback) and for studying auditory processing during both active task engagement and passive replay. While raw audio recordings are available via the ONE API, the NWB conversion focuses on capturing the scientifically critical timing information in the trials table and digital event channels. This design balances comprehensive data preservation with storage efficiency, allowing researchers to access detailed acoustic data when needed while maintaining fast access to behavioral and neural alignment information.

For most analyses involving audio timing and its relationship to neural activity, the NWB files contain sufficient information. For specialized acoustic analyses (e.g., studying ambient noise, animal vocalizations, or detailed spectral features), researchers should access the raw FLAC files and computed spectrograms via the ONE API.
