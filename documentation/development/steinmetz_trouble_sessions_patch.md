# Steinmetz NR_0029 Trouble Sessions (Neuropixels 2.4)

## Sessions

Four sessions from subject NR_0029 (steinmetzlab) fail raw conversion due to
corrupted SpikeGLX `.meta` files. All were recorded with Neuropixels 2.4 probes
and post-processed by an IBL tool that rewrote the metadata incorrectly.

Each session has a single probe insertion (different brain location each day,
different PID) with 96 neural channels from two shanks:

| EID | Date | Probe | PID | snsSaveChanSubset | Brain regions |
|-----|------|-------|-----|-------------------|---------------|
| `d85c454e-8737-4cba-b6ad-b2339429d99b` | 2023-08-29 | probe00a | `fb962a77...` | `0:47,96:143,384` | Hypothalamus, Thalamus |
| `de905562-31c6-4c31-9ece-3ee87b97eab4` | 2023-08-31 | probe00b | `6a7544a8...` | `48:95,144:191,384` | Midbrain, Subst. nigra ret. |
| `3a3ea015-b5f4-4e8b-b189-9364d1fc7435` | 2023-09-05 | probe00a | `bf746764...` | `0:47,96:143,384` | Hypothalamus, Thalamus |
| `4e560423-5caf-4cda-8511-d1ab4cd2bf7d` | 2023-09-07 | probe00a | `703f2cd2...` | `0:47,96:143,384` | Subst. nigra ret. |

All sessions have full modalities: AP + LF electrical series, NIDQ sync events,
spike sorting (151-326 units), video (left + right cameras, some have body),
pose estimation, pupil tracking, motion energy, wheel, lick times, and passive
protocol.

## Background: NP 2.4 per-shank splitting

NP 2.4 probes have 4 shanks but SpikeGLX records all shanks into a single
combined AP file (384 channels). NP 2.4 does **not** produce a native LF band
(unlike NP 1.0).

An IBL post-processing tool:

1. **Splits the AP data per shank** into separate binaries (`probe00a` for
   shank 0, `probe00b` for shank 1, etc.)
2. **Rewrites the AP `.meta`** to reflect only that shank's channels
   (marks `original_meta=False`)
3. **Generates LF from AP** by filtering/downsampling, creating `.lf.cbin` +
   `.lf.meta` from scratch

For BWM, only the spike-sorted shank was uploaded to the server. The file
structure on the server is:

```
raw_ephys_data/
    _spikeglx_ephysData_g0_t0.nidq.cbin
    _spikeglx_ephysData_g0_t0.nidq.ch
    _spikeglx_ephysData_g0_t0.nidq.meta
    probe00a/                                       <-- only the spike-sorted shank
        _spikeglx_ephysData_g0_t0.imec0.ap.cbin
        _spikeglx_ephysData_g0_t0.imec0.ap.ch
        _spikeglx_ephysData_g0_t0.imec0.ap.meta     <-- REWRITTEN by IBL tool
        _spikeglx_ephysData_g0_t0.imec0.lf.cbin    # GENERATED by IBL tool
        _spikeglx_ephysData_g0_t0.imec0.lf.ch
        _spikeglx_ephysData_g0_t0.imec0.lf.meta     <-- CREATED by IBL tool
```

The binary data is correct. Only the metadata is broken.

## Problem 1: Missing `~` prefix on list-type fields

SpikeGLX uses a `~` prefix to mark fields whose values are parenthesized lists.
The rewriting tool stripped this prefix.

| Field | Expected | Corrupted |
|-------|----------|-----------|
| Channel map | `~snsChanMap=(384,0,1)(AP0;0:0)...` | `snsChanMap=(384,0,1)(AP0;0:0)...` |
| Gain table | `~imroTbl=(24,384)(0 0 0 0 0)...` | `imroTbl=(24,384)(0 0 0 0 0)...` |
| Shank map | `~snsShankMap=(4,...)...` | `snsShankMap=(4,...)...` |

**How it breaks:** Neo's `read_meta_file()` only parses list values when the key
starts with `~`. Without it, the value is stored as a plain string. Iterating
over a string yields individual characters instead of channel entries:

```
ValueError: signal_channels do not have unique ids for stream 0
```

**Fix:** Prepend `~` to any bare `snsChanMap`, `imroTbl`, or `snsShankMap` key.

## Problem 2: LF meta `fileName` points to AP binary

The LF `.meta` was generated by copying the AP `.meta` and changing only four
fields (`acqApLfSy`, `fileSizeBytes`, `imSampRate`, `snsApLfSy`). The
`fileName` field was left unchanged:

```
# In .lf.meta:
fileName=.../_spikeglx_ephysData_g0_t0.imec0.ap.bin   # should be .lf.bin
```

**How it breaks:** Neo uses `meta["fileName"]` to determine stream type. When
both AP and LF metas claim to be AP streams:

```
KeyError: "key (0, 'imec0.ap') is already in the signals_info_dict"
```

**Fix:** In `.lf.meta` files, replace `.ap.bin` with `.lf.bin` in the
`fileName` field.

## Problem 3: Rewritten `snsSaveChanSubset`

The tool changed `snsSaveChanSubset` from the original non-contiguous range
(`0:47,96:143,384`) to a contiguous range (`0:96`).

The tool stored the original value in a backup field:
`snsSaveChanSubset_orig=0:47,96:143,384`.

**How it breaks:** Neo uses `snsApLfSy` to count channels (96), but
probeinterface uses `snsSaveChanSubset` to slice probe contacts and returns 97.
The mismatch causes a shape error:

```
TypeError: Can't broadcast (100, 96) -> (100, 97)
```

**Fix:** Restore `snsSaveChanSubset` from `snsSaveChanSubset_orig` when the
backup field exists.

## Problem 4: Oversized `snsChanMap` and `snsShankMap`

These fields should list only saved channels (matching `nSavedChans`). The
rewriting tool dumped the full 384-channel probe geometry instead. When neo
takes the first `nSavedChans` (97) entries from the 385-entry map, the 97th
entry is `AP96` instead of `SY0` (the sync channel):

```
Exception: set_probe(inplace=True) must have all channel indices
```

**Fix:** Trim `~snsChanMap` and `~snsShankMap` to only include entries for
channels listed in `snsSaveChanSubset`.

## Problem 5: `inter_sample_shift` IndexError (spikeinterface)

Not a corruption in the `.meta` files but a bug in spikeinterface triggered by
the non-contiguous `snsSaveChanSubset` used by NP 2.4 probes.

Spikeinterface computes `inter_sample_shift` by indexing into the full
`imroTbl` using channel indices from `snsSaveChanSubset`. With a non-contiguous
subset, some indices exceed array bounds:

```
IndexError: index out of bounds
```

**Fix:** Spikeinterface PR #4317 (merged, awaiting release > 0.103.2). We pin
spikeinterface to `main` in `pyproject.toml`. Additionally, both
`IblSpikeGlxConverter` and `IblNeuropixels2Converter` delete the
`inter_sample_shift` property from recording extractors before writing.

## Problem 6: Duplicate electrode table rows (channel_name mismatch)

When pre-populating the electrode table with anatomical localization before
neuroconv writes the electrical series, the `channel_name` values must match
what neuroconv will generate. Neuroconv deduplicates electrodes using a
virtual ID: `{group_name}_{electrode_name}_{channel_name}`.

For NP 2.4 with `snsSaveChanSubset=48:95,144:191,384`, the actual SpikeGLX
channel numbers start at 48, not 0. If the pre-populated table uses 0-based
numbering (`AP0,LF0`) while neuroconv uses the real numbers (`AP48,LF48`),
the virtual IDs don't match and neuroconv adds 96 duplicate rows without
localization.

**Result:** 192 electrode rows (96 with brain regions but unused, 96 without
brain regions but linked to the ElectricalSeries).

**Fix:** Parse `snsSaveChanSubset` from the `.meta` file and use the actual
channel numbers for `channel_name` generation. See `_get_saved_channel_numbers()`
and `add_probe_definition_to_nwbfile()` in `electrodes.py`.

## Implementation

### Patch modules

- `src/ibl_to_nwb/conversion/spikeglx_patches.py`: fixes corruptions 1-4
  (idempotent, scans all `.meta` files under a folder)
- `src/ibl_to_nwb/utils/electrodes.py`: `_get_saved_channel_numbers()` for
  electrode deduplication fix

### Where patches are applied

**1. Decompressed ephys folder** (`session.py`, after decompression):
Patches the files that neo's SpikeGLX reader uses. See `convert_session()`
around line 275.

**2. On-the-fly in electrode table creation** (`electrodes.py`, inside
`add_probe_electrodes_with_localization`): After ONE resolves the `.meta` file
but before `probeinterface.read_spikeglx()` reads it. Covers both raw and
processed conversion paths. See around line 569.

### Why two patch sites

The ONE API validates cached files by size and hash. Patching a file in the ONE
cache triggers re-download of the corrupted original. So:

- Decompressed folder patch handles neo's reader (raw conversion)
- On-the-fly patch handles probeinterface reads (both raw and processed)

Both are idempotent and no-ops for sessions with correct metadata.

### Upstream fixes

**python-neo:** Prefer actual file path over `fileName` for stream
identification; auto-detect list-shaped values without `~` prefix;
cross-check `snsApLfSy` against stream type.

**spikeinterface:** PR #4317 (merged, awaiting release > 0.103.2).
